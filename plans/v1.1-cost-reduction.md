# DUUL v1.1 — Reviewer Cost Reduction Plan

Hand-off doc for the implementing agent. Scope: reduce average reviewer cost per call before broad release. Out of scope: feature additions, schema changes that break v1 tool contracts.

## Context

`duul-tokens --plan max20 --all-time` snapshot (dev usage on this repo, 2026-04-14):

| metric | plan_review | code_review |
|---|---|---|
| calls | 117 | 61 |
| avg tokens / call | 46,271 | 77,429 |
| avg cost / call | $0.10 | $0.16 |
| total | $11.91 | $9.74 |

Provider currently in use: `gpt-5.4` ($2 / $8 per 1M tokens). Target: cut avg `code_review` token usage by **~40%** and avg cost across both tools by **~50%**, without degrading review quality.

The four levers are independent — implement and measure one at a time so we can attribute the impact.

## Task 1 — OpenAI prompt caching for the reviewer system prompt

**Files:** [src/services/providers/openai.ts](src/services/providers/openai.ts), [src/services/providers/anthropic.ts](src/services/providers/anthropic.ts)

**Why:** The system prompt + tool schemas are identical across iterations of the same review session. Anthropic and OpenAI both bill cached input at ~10% of normal rate. Right now nothing in the codebase sets `cache_control` (Anthropic) or relies on the automatic prefix cache (OpenAI ≥1024 tokens). Estimated saving: **40–55%** on input cost when iterating ≥2 rounds.

**What to do:**
1. **OpenAI** — `gpt-5.4` and `gpt-4.1*` automatically cache prefixes ≥1024 tokens; the system message just needs to be **stable across calls in the same review_id session**. Audit `OpenAIProvider` and ensure:
   - System prompt construction is deterministic (no timestamps, no per-call dynamic prefix).
   - The optional `prompt_cache_key` request param is set to the review session identifier so OpenAI can route to a warm cache.
   - Read `usage.prompt_tokens_details.cached_tokens` from the response and surface it in `TokenUsage` (extend `src/services/providers/types.ts` with `cached_input_tokens?: number`).
2. **Anthropic** — add `cache_control: { type: 'ephemeral' }` to the system prompt block and to large static tool schemas. Surface `usage.cache_read_input_tokens` and `usage.cache_creation_input_tokens` into `TokenUsage` (already partial — verify).
3. **Pricing** — update [src/services/pricing.ts](src/services/pricing.ts) `estimateCost` to subtract cached tokens from the input bucket and add a separate `cache_read` line at 0.1× input price.

**Acceptance:**
- Run `duul-tokens --all-time --json` before and after a fresh DUUL session of equivalent scope. `cached_input_tokens` on the 2nd+ call of a session must be > 0.
- Avg `plan_review` cost drops ≥30% over a 3-round ping-pong.

## Task 2 — Tighten default reviewer file-read scope

**Files:** [src/tools/code-review.ts](src/tools/code-review.ts), [src/schemas/code-review.ts](src/schemas/code-review.ts), reviewer prompt builders in [src/services/reviewer.ts](src/services/reviewer.ts)

**Why:** `code_review` averages 77k tokens, ~30k more than `plan_review`. Most of that delta is the reviewer pulling in adjacent files via `read_file` / `list_directory` exploration. Today `changed_files` is optional and the reviewer is free to roam.

**What to do:**
1. Add a hard cap on bytes the reviewer can read per session — config `DUUL_MAX_REVIEWER_BYTES` (default `200_000`). Track total bytes returned by `read_file` / `read_file_range` / `read_json` in the tool dispatcher. Reject further reads after the cap and tell the model in the tool result: `"reviewer file budget exhausted; rely on context already gathered"`.
2. When `changed_files` is provided, restrict `read_file` to: changed files + files referenced by changed files at depth 1 (imports/exports). Reject with a clear error otherwise — force the caller (Phase 2 author) to be explicit.
3. Update the reviewer system prompt to instruct: *"Prefer the diff and the explicit `changed_files` list. Only read additional files if the diff is genuinely insufficient to assess the blocking issue."*

**Acceptance:**
- New unit test confirms the byte cap is enforced (mock provider, count tool calls).
- Re-running a representative DUUL code_review cycle drops avg input tokens ≥25%.
- README documents the env var and the new `changed_files` requirement.

## Task 3 — Per-tool model override (no plan downgrade)

**Files:** [src/services/reviewer.ts](src/services/reviewer.ts), [src/schemas/common.ts](src/schemas/common.ts), [README.md](README.md), [README.ko.md](README.ko.md)

**Why:** Today `reviewer_config.model` is one global value. Some users want different models per phase — most commonly they want to **escalate code_review to a stronger model** while keeping plan_review on the same strong default. Expose the knob; do **not** ship a default that weakens plan_review.

**Architectural rationale:** Plan-phase defects compound through implementation. The whole value of DUUL is catching them early, so the default reviewer for plans must remain a strong model (current `gpt-5.4` or equivalent). The override exists for flexibility, not to push users toward cheaper plan reviews.

**What to do:**
1. Allow `reviewer_config.model` to be either a string (current) **or** an object `{ plan?: string; code?: string; partition?: string }`. Resolve in `getProvider()` based on the calling tool name. Backwards-compatible — string still works.
2. README section: "Per-tool model override" with a sample for users who want to spend MORE on code_review (e.g., `claude-opus-4` for code_review while plan_review stays on `gpt-5.4`). Explicitly note: **do not weaken plan_review** — plan defects compound.
3. Do NOT add cheap-model defaults or recommend mini/haiku for plan_review anywhere in the README.

**Acceptance:**
- Setting `reviewer_config.model = { plan: "gpt-5.4", code: "claude-opus-4" }` routes correctly per call (verified in usage log `model` field).
- Existing string-form configs still pass type checks and work unchanged.
- README mentions stronger-for-code use case, no cheap-plan recommendations.

## Task 4 — Surface iteration cost in tool responses

**Files:** [src/tools/plan-review.ts](src/tools/plan-review.ts), [src/tools/code-review.ts](src/tools/code-review.ts), [src/services/review-limits.ts](src/services/review-limits.ts)

**Why:** Right now the calling agent has no visibility into accumulated cost during a ping-pong. If round 4 happens, the agent doesn't know it just spent $0.50. A soft warning at round 3+ encourages the agent to either accept REVISE-with-minor-issues or escalate to human.

**What to do:**
1. Add `cumulative_cost_usd` and `cumulative_tokens` fields to the review response schema (optional — only populated if a session-level accumulator is wired up; otherwise omit).
2. When `iteration_count >= ceil(iteration_limit * 0.6)`, include a `cost_warning: "..."` field in the response (e.g., *"3rd plan review iteration; consider accepting current verdict or escalating."*).
3. Document the warning in CLAUDE.md so the orchestrating agent knows to react to it.

**Acceptance:**
- Manual test: run a 4-round plan ping-pong; `cost_warning` appears from round 3 onward.
- No breaking changes to existing schema consumers.

## Out of scope (do not touch)

- DUUL state file format (`.duul-state.json`) — unchanged
- MCP tool names / required argument names — unchanged
- usage-logger.ts behavior — already gated by `DUUL_DEBUG_TOKEN`

## Verification protocol

After all 4 tasks land:
1. Build: `npm run build` — no TS errors.
2. Run a fresh DUUL cycle on a sample task in a scratch repo.
3. `duul-tokens --since <start>` — compare averages against the baseline table above.
4. Pass criteria: `code_review` avg tokens ≤ 47k AND total avg cost/call ≤ $0.07.

## Suggested commit boundary

One PR per task. Each PR independently shippable and measurable.

## Follow-ups (out of v1.1 scope)

- **Budget-exhausted post-LLM gate:** when the reviewer's byte budget runs out mid-review, the final verdict may be under-informed. Add a gate that appends `"budget_exhausted"` to `gates_tripped` and forces `requires_human_review: true` (or downgrades APPROVE → REVISE). Low priority while the default cap is unset, since the gate would rarely trigger in practice.
